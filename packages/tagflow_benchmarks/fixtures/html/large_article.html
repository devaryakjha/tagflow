Native Runtime Field Guide

This fixture approximates a long-form release note, product brief, and implementation memo merged into one deterministic document.

1. Why this exists

Rich content in product surfaces often arrives from a CMS, AI response, or server-authored workflow. The renderer needs to stay predictable when the content tree becomes deep, styled, and repetitive.

The benchmark harness should measure parser cost on realistic content rather than isolated tags that never appear in production.

  • Repeat common prose structures.
  • Mix lists, code, and quotes.
  • Stay local-only and fully deterministic.

2. Product shape

Product-shaped rich content usually combines instructional copy, a few semantic headings, a short metrics table, and user-facing notes.

Benchmarks should answer whether the runtime feels acceptable on the fixtures that matter, not whether it can mimic a whole browser.

That means repeated sections are useful, because they create a document tree large enough to expose parser regressions without adding external variability.

3. Implementation notes

Parser work should remain separate from conversion and frame timing so each layer can move independently. 

const suite = TagflowParserBenchmarkSuite(
  warmupIterations: 5,
  sampleCount: 10,
);

Capturing medians, p95 values, and coefficient of variation makes the output more actionable than a single fastest run.

4. Repeated operational checklist

  1. Verify fixture validity before comparing renderers.
  2. Warm up the parser before taking timed samples.
  3. Store JSON output when a reviewable artifact is needed.
  4. Keep generated benchmark output out of version control.

The same checklist appears in several sections on purpose so the document remains long enough to exercise traversal and node reparenting paths.
